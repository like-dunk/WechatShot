# 扩展侧异常检测迁移指南

> **写给**：Chrome 扩展开发者（AI 同事）
> **日期**：2026-06-05
> **背景**：标题异常检测已从扩展侧迁移到 Reflo 后端 `/api/v1/plugin/release-info/batch` 接口，后端在每个 item 的 `data` 中新增了 `anomaly` 字段。扩展侧只需读取该字段，删除本地检测代码。

---

## 1. 后端接口变更

### 响应新增字段

`POST /api/v1/plugin/release-info/batch` 返回的每个 item 的 `data` 中新增：

```json
{
  "data": {
    "title": "华为乾崑智驾技术加持，奕境X9大六座SUV实力出众",
    "account": "奕境汽车",
    "platform": "视频号",
    "anomaly": {
      "detected": false,
      "confidence": 1.0,
      "reason": null
    }
  }
}
```

### `anomaly` 字段定义

| 字段 | 类型 | 说明 |
|------|------|------|
| `detected` | `boolean` | 是否检测为异常标题 |
| `confidence` | `float` | 置信度（0.0 ~ 1.0） |
| `reason` | `string \| null` | 异常原因，正常时为 `null`；异常时如"竞品瑞虎8" |

### 无标题的项

如果某条链接获取不到标题（失败或无 data），`anomaly` 字段为 `null`（不存在）。

## 6. 常见问题

**Q: 为什么不在 `applyRefloCorrection()` 内部处理 anomaly？**
A: `applyRefloCorrection()` 的职责是 Excel 字段纠错（对比本地和 Reflo 数据差异）。anomaly 是独立于字段纠错的检测结果，放在调用侧更清晰。

**Q: `settings.detectAnomalousTitle` 变量删除后，旧的存储数据会不会有问题？**
A: 不会。`chrome.storage.local` 中残留的旧 key 不会被读取，不影响功能。如果想清理，可在初始化时调用一次 `chrome.storage.local.remove(['detectAnomalousTitle', 'llmEnhance', 'llmModel', 'encryptedApiKey', 'apiKeyIv', 'apiKeySalt'])`。

**Q: 后端异常检测失败怎么办？**
A: 后端 LLM 调用失败时会降级到纯统计规则结果，不会导致接口报错。如果整个接口调用失败（网络问题），`refloData` 为 `null`，anomaly 检测自然跳过——和现有的 Reflo API 失败处理一致。
