# 后台播放数据截图 - 设计方案

日期：2026-07-14

## 1. 背景与目标

PPT 模版生成目前已支持"粉丝量截图"作为可选辅助图（来源：Excel/WPS 内嵌图片兜底提取 + 手动 ZIP/文件夹上传，按 `序号_昵称` 与任务匹配，仅在 `release-info-screenshot`/`dawanqu` 两种模式下渲染进 PPT）。

现需新增第二种同类辅助图——"后台播放数据截图"，处理方式与粉丝量截图完全对称，唯一区别是它是**独立的第二个图片来源**，两者可以同时存在、也可以任意一个单独存在。

## 2. 已确认的关键决策

| 决策点 | 结论 |
|---|---|
| 图片来源 | 与粉丝量截图完全一致：Excel/WPS 内嵌图片兜底提取 + 手动 ZIP/文件夹上传 |
| Excel 表头识别关键词 | 模糊匹配包含"后台播放"四字（与现有"播放量"数字指标列区分度高，且 `resolveXxxColumnIndex` 本身按"实际承载内嵌图片最多的列"择优，天然过滤纯数字列） |
| 适用 PPT 模式 | 仅 `release-info-screenshot`（发布信息截图单图单页）与 `dawanqu`（大湾区崭新模版）——与粉丝量截图适用范围完全一致；`clippings`/`link-screenshot` 不支持 |
| 三图排版（主图+粉丝图+播放图同时存在） | 主图占左半区；右半区上下堆叠（粉丝量图在上，播放数据图在下）。仅一个辅助图时保持现状（辅助图铺满右半区）；无辅助图时主图铺满整个原始图框 |
| 自定义/大湾区模板文件 | 不需要重新制作，代码在运行时对已有的"辅助图占位框"（或计算出的右半区）自动做二等分，现有仅含 2 个占位图形的模板文件无需改动即可直接支持 |
| 代码组织 | 缓存模块与 UI 状态层按现有"每种辅助图一个平行模块"的项目风格独立新建（`playback-source-cache.js`，与 `fan-source-cache.js` 结构对应）；`序号_昵称` 匹配算法与排版分栏计算这两块高风险逻辑抽成通用函数，粉丝图与播放图共用同一份实现 |

## 3. 架构总览

### 3.1 新增文件
- `playback-source-cache.js`：IndexedDB 缓存模块，逐行对应 `fan-source-cache.js`
  - 库名 `shipinhao-playback-source-cache`，store `playbackSources`（keyPath `id`），索引 `createdAt`，3 天 TTL
  - API：`putPlaybackSource(record) / getPlaybackSource(id) / deletePlaybackSource(id) / cleanupOldPlaybackSources(maxAgeMs)`
  - 挂载：`self.PlaybackSourceCache` / `window.PlaybackSourceCache`
  - 记录结构：`{ id: "playback-<ts>-<rand>", name, files: [{fileName, blob}], createdAt }`

`resolvePlaybackImagesForPpt` 优先级与 `resolveFanImagesForPpt` 完全对称：
1. `options.playbackImages` 已是数组直接用

## 4. pptx-builder.js 核心改动

### 4.1 匹配算法通用化（方案C）

现有 `parseFanImageKey`/`normalizeFanMatchKey`/`normalizeFanSequence`/`getTaskFanSequence`/`findTaskForFanParsed`/`buildFanImageMap`/`resolveFanImageForTask`/`resolveFanImageFromImageName`（第 622-702 行）均为纯算法、无 fan 语义耦合，重命名去掉 `Fan` 字样：`parseAuxImageKey`/`normalizeAuxMatchKey`/`normalizeAuxSequence`/`getTaskAuxSequence`/`findTaskForAuxParsed`/`buildAuxImageMap`/`resolveAuxImageForTask`/`resolveAuxImageFromImageName`，行为完全不变，仅重命名。

`loadFanImagesFromCacheRecord`（第 711-719 行）同理重命名为 `loadImagesFromCacheRecord`（函数体本就与"fan"无关，仅做 blob→normalizeImage 转换），导出对象里对应 key 一并改名；popup.js/auto-ppt.js 里粉丝图和播放图的缓存记录加载都调用这一个函数。

`buildReleaseInfoScreenshotItems`（第 599-620 行）改为：
```js
function buildReleaseInfoScreenshotItems(images, tasks, options = {}) {
  const title = normalizeLinkScreenshotText(options.title) || RELEASE_INFO_DEFAULT_TITLE;
  const availableTasks = Array.isArray(tasks) ? tasks : [];
  const fanMap = buildAuxImageMap(options.fanImages || [], availableTasks);
  const playbackMap = buildAuxImageMap(options.playbackImages || [], availableTasks);
  return buildLinkScreenshotItems(images, availableTasks).map((item) => {
    const task = item.task || {};
    const releaseInfo = task.releaseInfo || {};
    const fanImage = item.task
      ? resolveAuxImageForTask(item.task, fanMap) || resolveAuxImageFromImageName(item.image, fanMap)
      : null;
    const playbackImage = item.task
      ? resolveAuxImageForTask(item.task, playbackMap) || resolveAuxImageFromImageName(item.image, playbackMap)
      : null;
    return { image: item.image, fanImage, playbackImage, title, /* ...其余字段不变... */ };
  });
}
```

### 4.2 修正：图片位置不能再用数组下标 1 隐式代表"粉丝图"

现状 `imageLinks[1]` 硬编码代表粉丝图（第 936、969 行），因为过去辅助图永远只有 0 或 1 张。现在辅助图可能是"仅粉丝""仅播放""两者都有"三种情况之一，**继续用位置下标 1 会在"仅播放、无粉丝"时把播放图错误渲染成粉丝图占位**——这是本次改动必须避免的正确性问题，而不是可选的代码整洁问题。

修正方案：`buildReleaseInfoScreenshotPptx`（第 782-850 行）内部把 `imageLinks` 从位置数组改为带 `kind` 标记的对象集合：
```js
const imageLinks = { main: registerMedia(item.image) };
if (item.fanImage) imageLinks.fan = registerMedia(item.fanImage);
if (item.playbackImage) imageLinks.playback = registerMedia(item.playbackImage);
// registerMedia 内联原 807-816/817-827 行的 media 文件写入 + relId 占位逻辑，返回 {...image, relId:"", target}
```
`buildReleaseInfoSlideRelationships(baseRelsXml, images)`（第 852-868 行）入参改为 `Object.values(imageLinks)`（本就按数组遍历注册 relId，无需改动内部实现）。

`buildReleaseInfoScreenshotSlideXml`/`buildReleaseInfoTemplateSlideXml` 签名从 `(baseSlideXml, imageLinks, item)` 的 `imageLinks[0]/imageLinks[1]` 改为 `imageLinks.main/imageLinks.fan/imageLinks.playback`，按 4.3 节的布局规则渲染。

### 4.3 排版计算通用化

新增 `splitBoxVertically(box)`（镜像现有 `splitBoxHorizontally`，对半分 `cy` 而非 `cx`）：
```js
function splitBoxVertically(box) {
  const halfCy = Math.round(box.cy / 2);
  return [
    { x: box.x, y: box.y, cx: box.cx, cy: halfCy },
    { x: box.x, y: box.y + halfCy, cx: box.cx, cy: box.cy - halfCy },
  ];
}
```

新增 `layoutAuxImages(auxImages, auxBox)`，`auxImages` 是按 `[fanImage, playbackImage].filter(Boolean)` 顺序（固定粉丝在上、播放在下）：
```js
function layoutAuxImages(auxImages, auxBox) {
  if (auxImages.length <= 1) return auxImages.map((image) => ({ image, position: fitPictureIntoBox(image, auxBox) }));
  const [topBox, bottomBox] = splitBoxVertically(auxBox);
  return [
    { image: auxImages[0], position: fitPictureIntoBox(auxImages[0], topBox) },
    { image: auxImages[1], position: fitPictureIntoBox(auxImages[1], bottomBox) },
  ];
}
```

**内置模板路径**（`buildReleaseInfoScreenshotSlideXml`，原第 923-949 行）：
```js
const mainImage = imageLinks.main;
const auxImages = [imageLinks.fan, imageLinks.playback].filter(Boolean);
if (auxImages.length) {
  const [leftBox, rightBox] = splitBoxHorizontally(RELEASE_INFO_LAYOUT.image);
  spTree.appendChild(createPictureNode(doc, nextShapeId, item.account || "发布截图", mainImage.relId, fitPictureIntoBox(mainImage, leftBox)));
  nextShapeId += 1;

    const label = image === imageLinks.fan ? "粉丝量截图" : "后台播放数据截图";

    nextShapeId += 1;

} else {
  spTree.appendChild(createPictureNode(doc, nextShapeId, item.account || "发布截图", mainImage.relId, buildReleaseInfoPicturePosition(mainImage)));
}
```
无辅助图时行为与现状完全一致（主图铺满整框）；仅一个辅助图时 `layoutAuxImages` 返回单元素、`fitPictureIntoBox` 铺满 `rightBox`，与现状完全一致；两个辅助图时才走新增的上下二分支路。

**自定义/大湾区模板路径**（`buildReleaseInfoTemplateSlideXml`，原第 960-993 行）：
```js

const fanPlaceholder = pictures[1] || null; // 现有模板文件的第 2 个占位，不新增第 3 个占位
```

这段逻辑覆盖了 4 种组合（无辅助图 / 仅粉丝 / 仅播放 / 粉丝+播放），且"仅粉丝"这一现有行为的两个子分支（`fanPlaceholder` 存在用其自身几何框 / 不存在则对半分）保持不变，只是把"辅助图数量恒为 1"泛化成"辅助图数量为 1 或 2"。

## 5. 边界情况

| 场景 | 行为 |
|---|---|
| 只有播放图，没有粉丝图 | `item.fanImage` 为 `null`，`item.playbackImage` 有值；渲染时辅助区只放播放图一张，铺满辅助区（不再错误套用"粉丝图"位置/标签，见 4.2） |
| 播放图与粉丝图都没有 | 与现状完全一致，主图铺满整框 |
| 某个任务的截图没有匹配到 task（`item.task` 为空） | `fanImage`/`playbackImage` 均为 `null`（`resolveAuxImageFromImageName` 兜底逻辑要求先有 task 才尝试，与现状一致，不新增按截图文件名匹配无 task 项的行为） |
| `clippings`/`link-screenshot` 模式 | `resolvePlaybackImagesForPpt` 提前返回 `[]`，`buildPptByMode` 对应分支本就不展开 `pptOptions`，播放图完全不参与，零影响 |
| Excel 表头同时含"粉丝"和"后台播放"两列内嵌图片 | 两次独立调用 `resolveAuxColumnIndex`（各自用不同关键词），互不干扰，各自取"实际承载图片最多"的列 |
| 自动 PPT 场景弹窗已关闭 | `autoPptPlaybackSourceId` 随 `autoPptFanSourceId` 一起经 `chrome.storage.local` session 透传给 `auto-ppt.js`，逻辑对称 |
