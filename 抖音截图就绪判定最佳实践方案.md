# 抖音截图就绪判定最佳实践方案

> 2026-07-03 由多路 web 调研（10 方向）+ 关键论断对抗验证（6 条）+ 三方案独立设计 + 双评审打分产出。
> 状态：用户已确认全部默认决策点，2026-07-05 已按完整版实施完成（content.js + background.js，回归测试全绿，见文末实施记录）。

## 一、问题定义

抖音批量截图存在两类事故（其他平台无）：

- **事故 a（黑播放器）**：截图中播放器全黑、进度条 00:00/00:00、中央加载圆点、右侧推荐栏骨架屏。根因：`waitForVideoPlayback` 是软闸门（20s 超时返回 false 不报错），background 仅多睡 3 秒后照样截图。
- **事故 b（弹窗漏拍）**：残留登录/扫码弹窗偶尔漏进截图。根因：三轮弹窗移除结束后到快门之间有约 2-4 秒空窗，弹窗在此期间重新弹出。

## 二、调研核心结论（均附来源，已过对抗验证）

| 结论 | 置信度 | 对方案的影响 |
|---|---|---|
| 现有判据 `readyState>=2 && currentTime>0.5` 只证明"数据到了、时钟在走"，官方（WICG rVFC explainer）明确其不能证明像素已上屏 | 高 | 黑屏根因的理论解释 |
| `requestVideoFrameCallback`（rVFC，Chrome 83+）是唯一"帧已提交合成器"强信号，专为取代 currentTime 类启发式设计；muted autoplay 可靠触发 | 高（CONFIRMED） | 首帧硬闸门的核心信号 |
| 窗口被遮挡/tab 失活时 Chrome 停止渲染，rVFC 停摆 | 高 | 混跑批量需 `document.hidden` 暂停计时 |
| 视频 paused/无新帧时 rVFC 永不触发；`currentTime+=0.001` seek 可逼出一帧 | 高 | rVFC 子超时的逼帧补救 |
| 抖音网页版用字节 xgplayer + MSE（video src 为 blob:），容器状态类 `xgplayer-playing/isloading/nostart` 可作辅助信号；MSE 视频 canvas 采样不跨域污染（douyinvod CDN 实测还带 ACAO:*） | 中-高（CONFIRMED） | 辅助判据 + 像素采样备用通道 |
| 行业有截图后像素质检+自动重拍先例：Microlink 开源引擎（Sharp 抽样 ~25% 像素、阈值 245、通道极差<=8 判白屏后重拍）；Playwright 用连续截图比对稳定化；共识是坏图不能静默产出 | 高（原"无人做质检"论断被证伪） | 截图后纯色质检 |
| 风控：同 IP 立即 reload 同页是高危 bot 信号；固定 3s 任务心跳是自动化指纹；"停留过短"同样敏感（故否决主动压缩停留时长的方案） | 中（部分 UNCLEAR，取保守面） | 延后重试 + 随机抖动；不做激进提速 |

## 三、三方案评审结果

| 方案 | 评审1（工程正确性） | 评审2（风控/批量效率） |
|---|---|---|
| **minimal 最小增量双闸门** | **8.5（胜）** | **8.45（胜）** |
| defense-in-depth 四闸门纵深防御 | 7.3 | 7.8 |
| risk-control-first 信号前移快判延后 | 6.4 | 6.88 |

否决理由摘要：纵深防御 300-400 行、videoRect×zoom×dpr 坐标换算随 Chrome 版本漂移、黑帧阈值需样张标定，投入产出比低；风控优先派主动把单页停留压到 9 秒反而新开风控敞口，且波及四平台共享流程，回归风险最高。

## 四、最终方案：minimal + 双评审嫁接（约 100 行，零新权限、零新文件）

### 4.1 治黑屏（事故 a）

1. **`waitForVideoPlayback` 升级为三重判定**（content.js，500ms 轮询、总超时 20s 不变）：
   - 数据级（沿用）：`readyState>=2 && videoWidth>10 && currentTime>0.5`；
   - xgplayer 状态类（仅辅助）：`.xgplayer` 容器含 `xgplayer-playing` 且无 `isloading/nostart`；容器不存在则跳过；**rVFC 已触发且数据级通过、但状态类持续 5s 不满足时，以 rVFC 为准放行并记警告**（防抖音改版类名整批误杀）；
   - 合成级（新增硬判据）：一次 `requestVideoFrameCallback` 回调，3s 子超时；子超时后 `currentTime+=0.001` 逼帧，`seeked` 后再等 rVFC 2s；API 缺失降级为前两项。
   - `document.hidden===true` 期间暂停 20s 计时（混跑时非抖音快门抢占 active tab 导致 rVFC 停摆的假超时防护）。
2. **超时不再兜底硬截**：`videoReady=false` 且为 /video/ 页时 prepare 返回 `ok:false`，经现有 `validatePreparedPage` 抛错、走现有 `attempts<2` 重试；删除 `DOUYIN_FALLBACK_RENDER_WAIT_MS` 兜底分支（死代码）。
3. **风控文案速判前移**：`hasDouyinRiskText` 检查提到 20s 首帧闸门之前，风控件 5-8s 快速止损。

### 4.2 治弹窗漏拍（事故 b）

4. **MutationObserver 弹窗哨兵**（content.js，约 15 行）：prepare 时安装（childList+subtree，幂等），新增节点命中现有 `shouldRemoveElement` 即移除，覆盖 prepare 结束到快门的全时间窗。
5. **快门前终检 `shipinhaoFinalSweep`**（content.js 新导出 + background.js `captureTabSerial` 调用，仅抖音）：激活 tab 成功后、`captureVisibleTab` 前执行——再扫一轮 `removeBlockingOverlays` → 双 `requestAnimationFrame`（确保移除已走完渲染管线）→ `shouldRemoveElement` 复扫；仍有阻挡则抛"快门前检测到残留弹窗"走重试。漏拍窗口从约 2-4s 压缩到约 0.1s。

### 4.3 兜底与风控

6. **截图后纯色质检**（background.js service worker，`fetch(dataUrl)→createImageBitmap→OffscreenCanvas` 64px 降采样）：主色占比>99% 判纯色图；**灰度模式先只记日志不判死**，上线观察后再决定是否参与重试判定。不做 videoRect 裁剪（绕开 zoom/dpr 换算坑）。
7. **延后重试**：抖音判死任务置 `task.retryNotBeforeTs = now + 45s`，`nextTask` 过滤未到期项——消除"同 IP 立即 reload 同页"高危信号。
8. **任务间隔随机抖动**：`getTaskDelayMs` 抖音分支追加 `Math.random()*2000`，打破固定 3s 心跳指纹。
9. **prepare 前 `focusCaptureWindow` 补焦**（1 行）。
10. **类型化错误码**（`FIRST_FRAME_TIMEOUT` / `POPUP_PERSISTENT` 等附在 error message）：上线首周统计闸门超时率与终检抛错率，为后续收窄判据、下调 1500ms 沉降提供数据。

### 4.4 耗时与风控影响

- 正常件（>95%）：净增约 0.1-0.2s（rVFC 一帧 + 终检 50-100ms），批量平均基本不变。
- 黑屏异常件：原"25s 必出坏图" → 新"重试一次约 40-45s 出好图，或约 42s 明确 FAILED"。
- 风控暴露面：正常件零新增请求；重试不突破既有 attempts<2 预算且延后 45s；黑屏件在 prepare 即中止（省截图/下载/缓存），请求量净减。

## 七、实施记录（2026-07-05）

- 决策点确认：用户认可全部推荐默认值（完整版约 100 行 / 二次失败标 FAILED / 本次只覆盖 /video/ / 按混跑场景处理）。
- 按"先复现后修复"落地：
  - tests/douyin_first_frame_gate_repro.mjs（虚拟时钟驱动真实 content.js，9 场景）修复前 8 项 FAIL 复现事故，修复后全 PASS；
  - tests/douyin_capture_flow_repro.mjs（chrome 桩驱动真实 background.js，5 场景）修复前 6 项 FAIL 复现缺口，修复后全 PASS。
- 与设计的差异（实施中发现并修正的设计缺口）：
  1. 评审嫁接的"hidden 暂停计时 + prepare 前补焦一次"组合存在死锁隐患——混跑时非抖音快门抢占 active tab 后没有任何机制把抖音 tab 拉回前台，暂停计时会让 prepare 无限挂起。实施改为 background 侧 startDouyinVisibilityKeeper 周期守护（快门执行期间经 captureBusy 标志避让），并给 hidden 暂停设 60 秒总上限双保险。
  2. 纯色质检每通道量化到 16 级后再统计主色占比，规避 PNG 压缩噪声导致同色像素分散、阈值永不命中的问题。
- 变更明细见 docs/PROJECT_SUMMARY.md 变更记录（2026-07-05 条目）。

## 八、评审修复记录（2026-07-05）

实施后经多智能体评审（4 维度审查 + 对抗验证，13 项候选中 6 项确认、3 项证伪、5 项限流未验证由人工核对），确认并修复 4 个独立缺陷：

| 级别 | 缺陷 | 修复 |
|---|---|---|
| critical | 延后重试等待期 worker 零扩展 API 调用，MV3 SW 约 30 秒空闲回收会静默杀死整批运行（finalizeRun 永不执行，无失败清单/Excel/PPT） | __wait 分支每轮 chrome.storage.session.get 心跳重置空闲计时 |
| major | hidden 暂停按名义值记账：hidden 页定时器被 1 秒对齐导致少记一半；confirmFirstFrame 的 3+2 秒等待中被抢占零记账——混跑下健康视频可被误判 FIRST_FRAME_TIMEOUT 两次而 FAILED | 改用 visibilitychange 监听 + 实测累计（含进行中分段），60 秒上限保留 |
| minor | 页面刚重载时准备/终检函数缺失被静默放行（ok:true/clean:true），构成绕过全部闸门的黑图旁路 | prepareMissing/sweepMissing 显式标记，抖音件判失败走重试（[PREPARE_NOT_READY]/[SWEEP_NOT_READY]） |
| minor | 无 video 元素的 /video/ 页（作品失效提示/图集）被无差别判死约 85 秒零产出，丧失旧版可截失效凭证的能力 | 闸门加"确有 video 元素"前置条件，无播放器页放行留证 + [NO_VIDEO_ELEMENT] 观察日志 |

另按评审建议把弹窗哨兵降为纯选择器匹配（文案启发式弹窗由快门前终检兜底），避免抖音高频 DOM 变更页逐节点强制布局的卡顿风险。

被证伪不处理的 3 项：tab 销毁与 45 秒延时叠加（HEAD 行为一致，非新增回归）、终检双扫描拉长快门排队（量化主张失实，读写分离无 layout thrash）、rVFC 等待烧预算的具体时序主张（currentTime>0.5 时 rVFC 约一帧内即触发；其记账缺口已并入 major 项修复）。

回归：内容层 18 项 + 集成层 21 项断言全绿（新增 C5b 中途抢占、C10 无播放器留证、S6 重载旁路封死、S4 心跳存在性用例），窗口并发历史回归 5 场景全绿。

遗留观察（不属本次消缺，已知即可）：用户手动暂停（waitWhilePaused）超过 30 秒且 popup 关闭时，SW 同样可能被空闲回收——该风险在本次改动前即存在，如需彻底解决建议后续引入 chrome.alarms 调度。
